消息渲染管线重构 + 可视化能力跃升

这次更新的重点不是单点修补，而是把“Agent 如何输出一份可读的分析结果”整条链路重做了一遍。前台体验更顺，后台的会话、工具、类型和 Skill 体系也一起补齐。

概览

如果只看这一版最值得关注的变化，可以先看下面这六件事。

  • SVG 概念图上线：概念型问题不再只靠文字解释，Agent 可以直接输出安全过滤后的 SVG 示意图。
  • 消息融合渲染完成：文字和图表可以交替出现，最终效果更像图文分析报告，而不是工具结果堆叠。
  • 过程追踪更克制：中间推理默认折叠进“过程笔记”，最终回答延迟 450ms 渐进揭示，视觉更稳。
  • 图表交互增强：图表入场动画更平滑，点击时间点后还能自动生成追问继续诊断。
  • MCP 连接更聪明：闲聊和概念问答不再默认连工具服务，响应速度更快。
  • 平台能力一起补强：SSE 流、可视化组合引擎、Skill 诊断体系、类型系统与测试覆盖都同步推进。

核心体验更新

这一组变化最直接，用户在对话界面里基本都能感受到。

1. SVG 概念图：从 Mermaid 到矢量图

当用户问“峰谷套利原理是什么”“这个微网的拓扑结构”“排程决策流程”这类概念问题时，Agent 现在可以直接在对话里渲染 SVG 概念图。相比 Mermaid，这种方式对拓扑、功率流向、设备连接关系这类自由图形的表达力更强，也不会被固定图形语法绑住。

场景 示例问题 输出形式
原理解释 “峰谷套利 / 削峰填谷的原理是什么？” SVG 原理示意图
拓扑结构 “这个微网的设备连接关系” SVG 拓扑图
决策流程 “排程怎么决定充放电时段？” SVG 流程图
状态机 “电价时段是怎么划分的？” SVG 状态转换图
功率流向 “储能放电时功率怎么流动？” SVG 流向示意图
安全机制
  • 剥离 <script> 标签。
  • 移除 <foreignObject> 嵌入。
  • 清除所有 on* 事件处理器。
  • 替换 javascript: 协议 URL。
  • 替换 data: 协议 URL。

边界也被明确写进系统提示里了：功率曲线、SOC 变化、负荷趋势、电价走势这类时间序列数据禁止走 SVG，必须通过 MCP 拿数据后交给前端图表组件渲染。

2. 消息融合渲染：图文交替的报告式体验

以前的结果更像“先吐一大段文字，再把工具结果堆在底部”。现在渲染层把 Claude 返回的 content blocks 解析成统一的内容片段序列，按顺序交替展示文字和工具结果，阅读上更像一份结构化报告。

以前： [一大段分析文字...] [工具结果 1][工具结果 2][工具结果 3] 现在： [一段分析文字] [排程 vs 运行对比图表] [继续分析偏差原因] [电价时段图表] [最终结论和建议]
  • 内容解析：parseInterleavedContent() 统一产出 ContentSegment[]。
  • 交替渲染：完成态和流式态都按 segment 顺序渲染，文字和图表不再硬分家。
  • 组合检测保留：虽然图表视觉上分散，但组合逻辑仍能看到完整工具集合，不影响“三件套”“四件套”识别。

这里还顺手解决了一个细节问题：模型有时会先说半句话，再调工具，回来把这句话补完。现在只有恢复的文本明显像新的 Markdown 块时才会强制换段，避免一句话被机械拆开。

3. 过程追踪：折叠中间推理，最终回答渐进出现

复杂诊断时，Agent 的中间思路现在默认被收进“过程笔记”，折叠态只显示数量和最新预览；最终回答则在工具执行完成后延迟 450ms 显示，配合 shimmer 过渡，避免大段文字突兀弹出。

状态 何时出现 用户看到什么
none 普通对话，无需缓冲 文字直接显示
pre_tool Agent 开始输出文字但还没调工具 shimmer 占位动画
pending_final 工具全部执行完，最终回答即将出现 短暂 shimmer 后文字浮现

4. 图表交互增强：入场动画 + 点击钻取

图表不再是“数据一到就啪地出现”，而是增加了 180ms 的淡入和轻微缩放过渡。更实用的是点击钻取：用户可以直接点击图表中的某个时间段，系统会自动组装一条中文追问并发回 Agent，继续围绕这个时间点做诊断。

  1. 图表组件点击后传出分钟值和 HH:mm 时间字符串。
  2. 从当前工具输入提取 microGridId、emsId、date。
  3. 拼出诊断追问。
  4. 通过 gridpilot:send-query 自定义事件发回聊天输入框。

5. 聊天意图检测：闲聊加速，按需连接 MCP

“你好”“谢谢”“解释一下这个概念”这类不需要查数据的对话，现在不再一上来就连 MCP。系统用三层检测机制判断要不要连接工具服务，尽量在不漏判的前提下减少不必要的连接开销。

用户消息 → 第一层：是否系统自动触发（autoTrigger） → 第二层：是否命中 51 个领域关键词 → 第三层：历史对话中是否已有工具调用痕迹 → 如果都没有，直接跳过 MCP 连接

设计原则也很明确：宁可多连，不可漏连。误判成“需要工具”只是慢一点；误判成“不需要工具”会直接失去数据查询能力。

6. 微网可视化引擎：14+ 种图表组合布局

不同工具组合现在会自动匹配最合适的图表布局。拿到排程、运行、负荷、电价四类数据时，系统会优先用四合一布局；如果只拿到其中两三项，则切到更贴合的数据视图。

工具组合 布局名称 图表内容
排程 + 运行 + 负荷 + 电价 schedule-operation-with-load 排程计划 vs 实际运行 + 负荷叠加 + 电价背景色
排程 + 运行 + 电价 diagnosis 计划 / 实际对比 + 电价
排程 + 负荷 + 电价 schedule-with-load 排程计划 + 负荷预测 / 实际 + 电价
排程 + 电价 schedule-analysis 排程分析 + 电价时段
运行 + 负荷 operation-load 实际功率 + 负荷
单独排程 / 运行 / 负荷 / 电价 / 需量 / 回测 独立布局 各自对应的单一图表
  • 区间适配：运行数据 5 分钟颗粒度，图表统一到 15 分钟后再展示。
  • 字段兼容：同时兼容 camelCase 和 snake_case。
  • 历史 typo 容错：像 priceLcass 这样的后端旧字段名也在标准化层兼容掉了。

稳定性与底层能力

这部分不一定直接可见，但对长时对话、诊断稳定性和后续扩展都很关键。

7. SSE 流会话管理加固

复杂诊断经常要调 5 到 6 个工具，持续两分钟以上并不罕见。这一版把流管理器的空闲超时放宽到 330 秒，同时补上工具超时重试、请求注册表和结构化消息构建，减少“静默断流”和对话卡死。

长任务更稳

330 秒无新事件才中断，复杂诊断不容易被误伤。

工具超时可重试

保留已有上下文和内容，自动发起补跑，而不是整段对话作废。

结构化消息落库

流完成后把文字、工具调用和工具结果按序列化数组存库，支持后续融合渲染。

13 类事件统一派发

包括 text、tool_use、tool_result、status、error 等，流处理更清晰。

8. Skills 诊断体系整合

原先充电异常和放电异常分成两个 Skill，在跨充放电问题上天然容易只看到一半。现在统一成 ess-diagnosis，按同一条四阶段链路做诊断。

阶段一：上下文准备 阶段二：并行数据采集 阶段三：根因分析 阶段四：结构化报告
  • 共享模板：output-diagnosis.md、context-prep.md、unit-conversion.md 被提到 _shared/ 目录下复用。
  • response-confirm：压缩到 3 到 5 句，结论先行，不抢做原因分析。
  • response-review：偏差分析限制为 3 个最显著时段，并统一使用标准缩写。
  • revenue-analysis：明确区分真实收益与回测 / 模拟 / 计划收益，计算改为脚本执行。

9. 类型系统与数据模型扩展

这一版的功能跨度比较大，所以类型层也跟着补齐了，避免流式消息、结构化内容和多种 Agent 参数继续靠隐式约定往前推。

  • MessageContentBlock：统一表达 text / tool_use / tool_result / code。
  • StreamPhase：管理流生命周期状态。
  • SessionStreamSnapshot：记录工具列表、结果、token 用量、权限请求和最终消息。
  • StreamEvent：类型化事件通知。
  • ChatSession：新增 micro_grid_id / ems_id，把微网上下文持久化到会话。
  • ClaudeStreamOptions：扩展 autoTrigger、thinking、effort、agents 等能力。
